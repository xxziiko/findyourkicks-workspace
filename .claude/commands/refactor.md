이 턴의 산출물은 **리팩토링된 프로덕션 코드**다. 테스트는 리팩토링 전후 모두 통과해야 한다.
테스트 파일은 절대 건드리지 않는다.

## 입력

`$ARGUMENTS` — 리팩토링할 파일 경로 또는 영역 설명.

## 산출물

- 리팩토링된 프로덕션 코드
- 리팩토링 전후 테스트 결과 (둘 다 PASS)

## 프로세스

1. `$ARGUMENTS`에서 리팩토링 대상을 식별한다.
2. 리팩토링 **전에** 관련 테스트를 실행하여 green 상태를 확인한다:
   - Vitest: `pnpm --filter shop test -- --run`
   - Playwright: `pnpm test:admin` 또는 `pnpm test:shop`
3. 테스트가 green이 아니면 **중단하고 보고한다**. 깨진 코드는 리팩토링하지 않는다.
4. 리팩토링을 적용한다:
   - 중복 제거
   - 네이밍 개선
   - 복잡한 조건문 단순화
   - 공유 로직을 utils/hooks로 추출
   - 프로젝트 컨벤션에 맞게 정렬 (피처 슬라이스 구조, Biome 포매팅)
5. 리팩토링 **후에** 동일한 테스트를 실행하여 여전히 PASS인지 확인한다.
6. 결과를 보고한다: 변경 내용 + 테스트 결과.

## 규칙

- 테스트 파일을 수정하지 않는다.
- 새 기능을 추가하거나 동작을 변경하지 않는다 — 테스트가 수정 없이 통과해야 한다.
- public API 시그니처를 변경하지 않는다 (내부 전용이 아닌 한).
- 리팩토링 후 테스트 실패 시, 실패를 유발한 변경을 되돌리고 다른 접근을 시도한다.
- 관련 없는 코드를 리팩토링하지 않는다 — 대상 범위에만 집중한다.
